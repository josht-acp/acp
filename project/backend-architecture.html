Phase 2 Readiness · Backend Architecture · LP Portal

LP Portal — backend architecture.

Single-principal pragmatism. Institutional-grade posture. A self-hosted LP portal on a small VPS that a sole GP can operate credibly in front of Macquarie Pension Trust and the next ten LPs, and that scales cleanly to Fund II first close without a rebuild.

§1 · Executive summaryDecision in one page.

The portal ships on a single VPS in Sydney, Node + Postgres, behind Cloudflare. Magic-link auth, per-LP IP allowlists, document gating on the server, full audit log. A$40/month at launch, A$180–240/month through Fund II first close. No multi-region, no Kubernetes, no AWS console — because we don't need them and their operational weight costs more than their value at this scale.

Recommendation · build
Hetzner Cloud CX32 · Sydney region · Node 20 + Fastify + Postgres 16 + S3-compatible object storage, fronted by Cloudflare.
Four vCPU, 8 GB RAM, 80 GB NVMe. Dedicated PostgreSQL instance on the same host at launch, lifted to a managed instance at Fund II close. Object storage via Hetzner Storage Box for documents; Cloudflare R2 as the fallback. Magic-link auth via Resend. IP allowlisting per LP account. Full audit log on every read and write.

What this buys us

  • Credibility with MPT and institutional LPs — self-hosted, sovereign-data (Sydney), auditable, in-house log of every document access.
  • Speed to first LP — three weeks from decision to a gated production portal for one LP (MPT).
  • Clean scale path — every choice on this stack has a documented upgrade to Fund II scale without rewriting the app.
  • Sub-A$3k/year infrastructure cost through the first 15 LPs and A$200M AUM.

What this doesn't buy us

  • SOC2 Type II out of the box (we scope the 12-month roadmap to get there).
  • Nine-nines availability (we target 99.9% — one 4-hour maintenance window per quarter is acceptable for an LP portal).
  • A team of three engineers (we codify the runbook so one person can operate this).
Honesty caveat. Standing up a production portal with live LP data is a real engineering track — 2–4 weeks to first LP, then ongoing care. This document is the brief for that track. The lp-portal.html file is a front-end mock against seed data; it is not wired to any of the services described below. When you say go, we spin up a new project with a different posture (deploy pipeline, secret management, staging, backup drills).

§2 · VPSWhere the portal runs.

Three choices. All three give us a box in Sydney with NVMe and dedicated vCPU, which is what the portal actually needs. We pick on price per vCPU, BYO-ability of Postgres, and the vendor's posture toward institutional data.

Why Hetzner over AWS

At our scale — one Postgres instance, one app node, a few GB of documents — AWS charges a premium for elasticity we won't use. Hetzner's CX32 runs the entire portal at 10–15% CPU and gives us headroom for 50+ concurrent LPs. We pay for the power we use, not the power we might one day use. When Fund II lands and we need HA, we migrate to AWS Sydney with the same Docker image and Postgres dump.

Sovereignty check. MPT mandate compliance requires LP data to stay in Australia. Hetzner operates an AU1 region in Sydney — we confirm the region on account provisioning and document it in the LP DDQ responses.

§4 · Data modelNine tables. No more.

Every LP-visible page in the mock maps to a read over these tables. We resist the urge to over-model. We also resist the urge to push fund accounting into this schema — the portal is a read view; source of truth remains Xero + the GP accounting system, synced nightly.

Row-level isolation

Postgres RLS enforces that every read carries current_setting('app.account_id') and filters the row set. Application code sets this once per request after auth; a bug in a controller cannot leak an LP's data because the database rejects the read.

Fund accounting sync

The portal is not the book of record. A nightly worker pulls from Xero (AR, fee realisations) and the GP accounting file (valuations, call ledger) and upserts into Postgres. One-way. Portal cannot write fund figures back. This is the compliance story — the book of record never moves.

§5 · AuthHow LPs get in.

Magic-link, not passwords

Institutional LPs log in five times a quarter. A password they'd forget and reset each time is worse than a magic link to their signing email. We pair the link with:

  • Per-account IP allowlist — MPT accesses the portal from one or two office ranges. We codify them at onboarding. Login attempts from outside those ranges require a secondary email confirmation from the managing partner on the LP side.
  • Optional MFA — TOTP via Authy/1Password. Mandatory for operations-role users (those who can download documents in bulk).
  • Session binding to IP — sessions invalidate on IP change, preventing token replay.
  • 15-minute idle timeout · 8-hour hard timeout.

Login flow

  1. LP enters email at portal.acp.com
  2. Server checks email → lp_user row exists · account is active
  3. Server verifies request IP against account.ip_allowlist
        ├─ in range  → continue
        └─ out-of-range → log audit_event, email GP, require secondary approval
  4. Server issues a 32-byte single-use token, stores hash, TTL 10 min
  5. Server sends magic link via Resend; only to the signing email on file
  6. LP clicks → server verifies token + IP still in range + TTL
  7. Server issues session JWT (15 min idle, 8 hr hard cap), HttpOnly SameSite=Strict cookie
  8. Every subsequent request sets app.account_id in PG session → RLS enforced
  9. Every request → audit_event with verb, target, IP, user-agent

Document gating

Documents never stream through Fastify. On request, server:

  1. Validates the user can see document.id under visibility (join against their commitments).
  2. Writes a doc.download audit event including the target document SHA.
  3. Mints a pre-signed URL to object storage with a 5-minute TTL and Content-Disposition: attachment.
  4. Returns the URL. Client redirects. Storage serves bytes, logs a separate CF access log.

Two independent logs (application audit + CF access) for the same event make document forensics credible to an LP auditor.

Australian data sovereignty

  • All production data (Postgres, object storage, CF cache) provisioned in Sydney region.
  • Backups replicated to a second Sydney facility (Hetzner WAL-ship + Storage Box geo-redundant within AU).
  • No cross-border processing except Resend (US SMTP) — we document this in the DDQ and only send non-PII transactional email via it. Magic links include no PII in the URL.

Data retention

  • LP documents: retained for life-of-fund + 7 years.
  • Audit log: immutable, WORM-ed to object storage, 7-year retention.
  • LP user accounts: archived on LP exit, retained for 7 years, then purged.

The runbook

Written into ops/RUNBOOK.md alongside the repo. Covers: deploy, rollback, restore-from-backup, rotate secrets, revoke LP access in under 60 seconds, handle a suspected breach. One document. Updated on every material change. Reviewed quarterly.
